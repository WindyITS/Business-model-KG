# Reproducibility Guide

This is the reviewer-first guide. It explains what is present in a fresh clone,
what must be downloaded or generated, and which path to follow depending on
what you want to verify.

The project has three reproducibility islands:

1. use the project locally
2. reproduce the extraction evaluation
3. reproduce the query-stack fine-tuning workflow

They are connected by the same project story, but they are intentionally
separate. You can evaluate the extraction experiment without rerunning
fine-tuning, and you can use the runtime without rebuilding the local
router/planner bundle.

## Fresh Clone Artifacts

A fresh clone contains source code, prompt files, ontology files, example filing
texts, tests, benchmark files, and checked-in result reports. It does not
contain local generated outputs, large model bundles, or fine-tuning artifacts.

Those missing local folders are expected:

```text
outputs/
runtime_assets/query_stack/
data/query_planner_curated/
finetuning/artifacts/
```

They are ignored by Git because they are generated locally or downloaded from
public artifacts.

## Artifact Setup Matrix

| Artifact | Needed For | Source | Local Destination |
| --- | --- | --- | --- |
| Example 10-K text files | Running extraction examples | Tracked in Git | `data/*.txt` |
| Evaluation benchmark and generated extraction outputs | Reproducing extraction metrics without rerunning LLM extraction | `WindyITS/business-model-kg-benchmark-outputs` | `evaluation/benchmarks/` and `outputs/<company>/<pipeline>/latest/` |
| Query-stack bundle | Local routed natural-language querying | `WindyITS/business-model-kg-query-stack` or `publish-query-stack` | `runtime_assets/query_stack/` |
| Query-planner fine-tuning data | Rerunning router/planner fine-tuning | `WindyITS/business-model-kg-query-planner-data` | `data/query_planner_curated/` |
| Fine-tuning artifacts | Rebuilding the query-stack bundle | Generated by fine-tuning commands | `finetuning/artifacts/kg-query-planner/` |

Most root-level commands assume the main environment has been bootstrapped with
`./scripts/bootstrap_dev.sh`. Install the Hugging Face CLI in that environment
when downloading public artifacts from the root project:

```bash
./venv/bin/python -m pip install "huggingface_hub[cli]"
```

## Island 1: Use The Project Locally

This path verifies that the source checkout works as a local extraction and
runtime tool.

Bootstrap the main environment:

```bash
./scripts/bootstrap_dev.sh
```

Run a light readiness check:

```bash
./scripts/kg-health-check --skip-neo4j
```

Run an extraction without touching Neo4j. This requires a local
OpenAI-compatible endpoint such as LM Studio running at the default URL, or a
hosted provider/API key passed through the runtime options.

```bash
./scripts/kg-pipeline data/microsoft_10k.txt --skip-neo4j
```

This creates a saved run under `outputs/`. The output folder contains the
pipeline artifacts, resolved triples, validation report, and run summary.

If you want the graph database layer, start Neo4j and load saved outputs:

```bash
docker compose up -d
./scripts/kg-neo4j-load
./scripts/kg-neo4j-status
```

If you want the local routed query stack, download the published bundle:

```bash
./venv/bin/huggingface-cli download WindyITS/business-model-kg-query-stack \
  --local-dir runtime_assets/query_stack
```

Then install the optional query dependencies:

```bash
./venv/bin/python -m pip install -e ".[query-stack]"
```

The query commands can still use hosted fallback behavior without the local
bundle, but the local router/planner path needs `runtime_assets/query_stack/`.

## Island 2: Reproduce Extraction Evaluation

This path verifies the reported extraction metrics. It does not require
rerunning LLM extraction calls because the generated extraction outputs are
published.

From a fresh clone, bootstrap the main environment first:

```bash
./scripts/bootstrap_dev.sh
./venv/bin/python -m pip install "huggingface_hub[cli]"
```

Download the public evaluation artifact:

```bash
./venv/bin/huggingface-cli download WindyITS/business-model-kg-benchmark-outputs \
  --repo-type dataset \
  --local-dir hf_evaluation_artifacts
```

Install the benchmark files:

```bash
mkdir -p evaluation/benchmarks
rsync -a hf_evaluation_artifacts/benchmarks/ evaluation/benchmarks/
```

Install the generated outputs into the local layout expected by the evaluator:

```bash
./venv/bin/python - <<'PY'
from pathlib import Path
import shutil

source_root = Path("hf_evaluation_artifacts/outputs")
target_root = Path("outputs")

for source_path in source_root.glob("*/*/resolved_triples.json"):
    company = source_path.parts[-3]
    pipeline = source_path.parts[-2]
    target_path = target_root / company / pipeline / "latest" / "resolved_triples.json"
    target_path.parent.mkdir(parents=True, exist_ok=True)
    shutil.copy2(source_path, target_path)
PY
```

Run the evaluator:

```bash
for pipeline in zero-shot memo_graph_only analyst; do
  for split in dev test; do
    ./venv/bin/python -m evaluation.scripts.evaluate \
      --pipeline "$pipeline" \
      --split "$split" \
      --yes
  done
done
```

The generated reports are written under:

```text
evaluation/results/<pipeline>/<split>/
```

For scoring definitions and output-file details, see
[`evaluation.md`](./evaluation.md).

## Runtime Check: Query Public Artifacts

This path verifies the interactive graph layer using public generated outputs
instead of rerunning extraction.

First complete the artifact setup from Island 2 so public generated
`resolved_triples.json` files are installed under:

```text
outputs/<company>/<pipeline>/latest/
```

Then start Neo4j and load those saved outputs:

```bash
docker compose up -d
./scripts/kg-neo4j-load
./scripts/kg-neo4j-status
```

Download the published local query-stack bundle and install the optional query
dependencies:

```bash
./venv/bin/huggingface-cli download WindyITS/business-model-kg-query-stack \
  --local-dir runtime_assets/query_stack
./venv/bin/python -m pip install -e ".[query-stack]"
```

Render a Cypher query:

```bash
./scripts/kg-query-cypher "Which companies sell to developers through direct sales?"
```

Run the query against Neo4j:

```bash
./scripts/kg-query "Which companies sell to developers through direct sales?"
```

This path connects the public extraction artifacts, local Neo4j load, and local
query-stack bundle into one end-to-end runtime check.

## Island 3: Reproduce Query-Stack Fine-Tuning

This path verifies that the local query router/planner bundle can be rebuilt
from the published fine-tuning dataset.

The fine-tuning island has its own environment and package under `finetuning/`.
It is intentionally separate from the main extraction/runtime package.

This is the heaviest reproduction path. It assumes Apple Silicon for the MLX
planner path, network access for model downloads, several gigabytes of disk
space, and enough memory for local LoRA training. If you only want to use the
query runtime, download the published query-stack bundle instead of rerunning
fine-tuning.

Bootstrap only the fine-tuning environment:

```bash
bash finetuning/scripts/bootstrap_env.sh
```

Download the public query-planner dataset:

```bash
bash finetuning/scripts/download_query_planner_data.sh
```

Activate the isolated environment:

```bash
source finetuning/.venv/bin/activate
```

Run the workflow:

```bash
prepare-data
train-router
eval-router
train-planner
eval-planner
publish-query-stack
```

The workflow reads `finetuning/config/default.json`, writes training artifacts
under `finetuning/artifacts/kg-query-planner/`, and publishes the runtime bundle
to `runtime_assets/query_stack/`.

For hardware expectations, model notes, and fine-tuning details, see
[`../finetuning/README.md`](../finetuning/README.md).

## Checks

For a quick reviewer sanity check, use:

```bash
./scripts/kg-health-check --skip-neo4j
```

For the full maintainer check, use:

```bash
bash ./scripts/check_repo.sh
```

The full maintainer check runs the main test suite, fine-tuning tests,
compilation checks, command-wrapper checks, and package smoke installs.

## Installed Fine-Tuning Package Notes

The fine-tuning package can be installed outside the source checkout. In that
mode, it carries a fallback copy of the default config. Relative paths in that
config resolve from the current working directory unless
`KG_QUERY_PLANNER_FT_ROOT` is set.

Use this when running the fine-tuning CLIs from an installed wheel while
keeping data and artifacts in a project directory:

```bash
export KG_QUERY_PLANNER_FT_ROOT=/path/to/project-root
```
