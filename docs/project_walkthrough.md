# Project Walkthrough

This document explains the project in plain language. It is meant for a reader
who wants to understand the architecture before looking at commands or code.

## The Problem

Public filings describe how companies operate, but they do it in prose. That
prose is rich and legally precise, yet difficult to compare across companies.
One company may describe a business segment, another may describe products, and
another may describe customer channels or geographies in a different order.

Business Model KG turns selected 10-K business sections into a graph with a
consistent schema. The goal is not to preserve every sentence. The goal is to
extract a comparable business-model view that can be inspected, evaluated, and
queried.

## The Core Idea

The project keeps three pieces aligned:

- prompts and pipelines decide how the language model is asked to extract facts
- the ontology defines which node and relation types are allowed
- the validator keeps saved graph outputs inside that ontology

That means the output is not just free-form model text. Each run leaves behind
structured artifacts, resolved triples, and validation reports that can be
audited later.

## The Graph Shape

The graph is centered on business segments.

`Company` is the reporting company or another named company. `BusinessSegment`
is the main business-model anchor. `Offering` represents products, services,
platforms, applications, or named product families.

The main pattern is:

```text
Company -> HAS_SEGMENT -> BusinessSegment
BusinessSegment -> OFFERS -> Offering
BusinessSegment -> SERVES -> CustomerType
BusinessSegment -> SELLS_THROUGH -> Channel
Offering -> MONETIZES_VIA -> RevenueModel
Company -> OPERATES_IN -> Place
Company -> PARTNERS_WITH -> Company
```

Some node types, such as customer types, channels, and revenue models, use
closed canonical labels. This keeps outputs comparable across companies.

For the full schema and validation policy, see [`ontology.md`](./ontology.md).

## The Extraction Pipelines

The repository includes three extraction strategies:

- `analyst`: builds a structured analyst memo, augments it, compiles it into a graph, and critiques the graph
- `memo_graph_only`: builds the first analyst memo and compiles it into a graph without the later augmentation and critique stages
- `zero-shot`: asks directly for the ontology graph in a single pass

All three pipelines share the same downstream resolver, validator, output
layout, and evaluator. That keeps the comparison focused on extraction strategy
rather than hidden differences in post-processing.

## What A Run Produces

A successful extraction run is saved under:

```text
outputs/<company>/<pipeline>/latest/
```

The exact files depend on the pipeline, but the important artifacts are:

- intermediate model outputs such as memos or graph-compilation JSON
- `resolved_triples.json`, the graph used by evaluation and Neo4j loading
- `validation_report.json`, the validator's account of accepted and rejected triples
- `run_summary.json`, the summary of the run configuration and outcome

Failed runs are kept separately so a failed experiment does not silently replace
the last successful output.

## Evaluation

The evaluator compares saved `resolved_triples.json` files against curated gold
benchmark triples. It reports exact precision, recall, F1, macro-F1, and a
relaxed graph-aware F1.

The evaluation island is useful because it separates model extraction from
measurement. You can reproduce the reported evaluation from checked-in
reference outputs and the public benchmark mirror without rerunning every LLM
extraction call.

For the reproduction path, see [`reproducibility.md`](./reproducibility.md).
For metric details, see [`evaluation.md`](./evaluation.md).

## Runtime And Querying

Neo4j is optional. Extraction and evaluation can run without it.

When Neo4j is available, saved graph outputs can be loaded into a local graph
database. The query commands can then render read-only Cypher or execute a
natural-language question against the live graph.

The natural-language query runtime has two routes:

- a local router/planner bundle for supported graph-shaped questions
- a hosted fallback path for broader read-only Cypher generation

The local bundle is stored under `runtime_assets/query_stack/`. It is ignored
by Git because it is a model artifact, but it can be downloaded from Hugging
Face or generated by the fine-tuning island.

For command details, see [`runtime_guide.md`](./runtime_guide.md).

## Fine-Tuning Island

The fine-tuning workflow under `finetuning/` trains the local query router and
planner. It is deliberately isolated from the main runtime package.

The output of that island is a deployable query-stack bundle:

```text
runtime_assets/query_stack/
```

The main runtime consumes that bundle, but it does not import fine-tuning code.
This keeps the normal extraction/runtime project lighter while still preserving
the path used to reproduce the local query stack.

For details, see [`../finetuning/README.md`](../finetuning/README.md).

## A Good Mental Model

Think of the project as five layers:

1. prompts ask for business-model information
2. pipelines organize the extraction process
3. the ontology defines what counts as a valid graph
4. outputs preserve each run for inspection and evaluation
5. Neo4j and the query stack make saved graphs interactive

The repository is built so each layer can be inspected separately, while the
full workflow still works end to end.
